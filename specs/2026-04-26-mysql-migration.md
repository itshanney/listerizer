# Technical Design: SQLite → MySQL Migration

**Date:** 2026-04-26
**Status:** Draft
**Stack:** Java 25, Spring Boot 4, MySQL 9.2, HikariCP, Flyway, Hibernate 7 / Spring Data JPA
**Source Spec:** [features/2026-04-11-reading-list-rest-api.md](../features/2026-04-11-reading-list-rest-api.md)

---

## Overview

This design replaces the embedded SQLite file with a MySQL 9.2 server, adopts Flyway for schema lifecycle management, and replaces JdbcTemplate with Hibernate/Spring Data JPA for persistence. 
* Flyway continues to own all DDL; Hibernate runs in `validate` mode only and never touches the schema. 
* `Item` is converted from a Java record to a regular class and annotated as the JPA entity directly — no separate entity class is needed. 
* `ItemRepository` becomes a Spring Data `CrudRepository` interface, eliminating the previous `ItemJpaRepository` intermediary. 
* The insert-or-fetch try-catch logic moves from the repository to `ItemService`, whose `create` method must not be `@Transactional` so that `CrudRepository.save()` commits in its own transaction and surfaces a duplicate-URL `DataIntegrityViolationException` within the catch block. 
* The REST API surface and Java package structure are otherwise unchanged.

**Component responsibilities:**

| Component | Responsibility |
|---|---|
| `ItemRepository` | Spring Data `CrudRepository` interface; provides `save`, `findByUrl`, `findAllByOrderByIdAsc` |
| `ItemService` | Validates requests; owns the insert-or-fetch try-catch logic against `ItemRepository` |
| `Item` | JPA entity and domain object; owns column mappings and record-style accessors |
| Hibernate ORM | Object-relational mapping; runs in `validate` mode — reads schema, never modifies it |
| HikariCP | Manages the MySQL connection pool |
| MySQL 9.2 | Durable relational store |
| Flyway | Versioned schema migration tool; sole owner of DDL |

**Failure modes:**

| Component | Failure | Behavior |
|---|---|---|
| MySQL | Unreachable at startup | Spring Boot fails to start; systemd/Kubernetes restarts the pod |
| MySQL | Unreachable mid-flight | HikariCP connection acquisition times out; request returns `500` |
| MySQL | Disk full | Write fails; `DataIntegrityViolationException` → `500` (not a duplicate-URL conflict) |

## Data Model

Schema is unchanged from the MySQL target (Flyway V1 migration remains identical):

| Column | MySQL type | JPA mapping | Notes |
|---|---|---|---|
| `id` | `BIGINT AUTO_INCREMENT PRIMARY KEY` | `@Id @GeneratedValue(IDENTITY)` | Generated by the database |
| `url` | `VARCHAR(2048) NOT NULL` | `@Column(nullable = false, length = 2048)` | Uniqueness enforced by DB prefix index; no `unique = true` in entity to avoid validation conflict |
| `create_time` | `BIGINT NOT NULL` | `@Column(name = "create_time", nullable = false)` | Epoch seconds; stored as primitive `long` |

**Why no `unique = true` on `url`:** The DB has a prefix unique index `url(255)`, not a full-column unique index. Hibernate's schema validator would reject a mismatch between the entity annotation and the DB metadata. Uniqueness is enforced at the DB level by the Flyway migration; the entity annotation is omitted to avoid the discrepancy.

**`Item` as entity:** `Item` moves from an immutable record to a regular class with JPA annotations. Its accessors retain the record naming convention (`id()`, `url()`, `createTime()`) so no call sites outside the `repository` package require changes. The protected no-arg constructor satisfies JPA's reflection requirement and is not part of the public API.

**Full Flyway schema (unchanged):**

```sql
CREATE TABLE items (
    id          BIGINT        AUTO_INCREMENT PRIMARY KEY,
    url         VARCHAR(2048) NOT NULL,
    create_time BIGINT        NOT NULL,
    UNIQUE KEY url_unique (url(255))
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;
```

`open-in-view=false` avoids holding a database connection open for the full duration of an HTTP request.

### 3. Flyway migration — unchanged

**`src/main/resources/db/migration/V1__create_items_table.sql`** is identical to the original MySQL migration design.

### 4. Updated: `Item.java`

`Item` moves from a Java record to a JPA entity class. The `domain` package location is unchanged. Accessors keep the record naming convention so no call sites outside this class require modification.

**`src/main/java/dev/brickfolio/listerizer/domain/Item.java`**

```java
package dev.brickfolio.listerizer.domain;

import jakarta.persistence.*;

@Entity
@Table(name = "items")
public class Item {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 2048)
    private String url;

    @Column(name = "create_time", nullable = false)
    private long createTime;

    protected Item() {}

    public Item(String url, long createTime) {
        this.url = url;
        this.createTime = createTime;
    }

    public long id()        { return id; }
    public String url()     { return url; }
    public long createTime(){ return createTime; }
}
```

### 5. Updated: `ItemRepository.java`

`ItemRepository` becomes a public Spring Data interface extending `CrudRepository<Item, Long>`. The insert-or-fetch try-catch logic moves to `ItemService` (see section 6). `findAllByOrderByIdAsc()` is a derived query that returns results ordered by `id` without requiring `Sort` or `JpaRepository`.

**`src/main/java/dev/brickfolio/listerizer/repository/ItemRepository.java`**

```java
package dev.brickfolio.listerizer.repository;

import dev.brickfolio.listerizer.domain.Item;
import org.springframework.data.repository.CrudRepository;

import java.util.List;
import java.util.Optional;

public interface ItemRepository extends CrudRepository<Item, Long> {
    Optional<Item> findByUrl(String url);
    List<Item> findAllByOrderByIdAsc();
}
```

### 6. Updated: `ItemService.java`

`ItemService.create` gains the insert-or-fetch try-catch and must **not** be `@Transactional`. `CrudRepository.save()` is `@Transactional` internally (`SimpleJpaRepository`), so each call commits in its own transaction and flushes immediately. If `create` were `@Transactional`, `save()` would join the outer transaction and defer the flush to commit time — the `DataIntegrityViolationException` would be thrown after the catch block, not within it.

**`src/main/java/dev/brickfolio/listerizer/service/ItemService.java`**

```java
package dev.brickfolio.listerizer.service;

import dev.brickfolio.listerizer.api.ItemRequest;
import dev.brickfolio.listerizer.domain.Item;
import dev.brickfolio.listerizer.repository.ItemRepository;
import org.springframework.dao.DataIntegrityViolationException;
import org.springframework.stereotype.Service;

import java.net.URI;
import java.net.URISyntaxException;
import java.util.List;

@Service
public class ItemService {

    private final ItemRepository repository;

    public ItemService(ItemRepository repository) {
        this.repository = repository;
    }

    public InsertResult create(ItemRequest request) {
        validateRequest(request);
        try {
            Item saved = repository.save(new Item(request.url(), request.createTime()));
            return new InsertResult(saved, true);
        } catch (DataIntegrityViolationException e) {
            Item existing = repository.findByUrl(request.url())
                    .orElseThrow(() -> new IllegalStateException(
                            "URL conflict but no existing row found: " + request.url(), e));
            return new InsertResult(existing, false);
        }
    }

    public List<Item> list() {
        return repository.findAllByOrderByIdAsc();
    }

    private void validateRequest(ItemRequest request) {
        if (request == null) {
            throw new ValidationException("Request must be valid and non-empty");
        }
        if (request.url() == null || request.url().isBlank()) {
            throw new ValidationException("url is required and must be a valid URL");
        }
        try {
            URI uri = new URI(request.url());
            if (uri.getScheme() == null || uri.getHost() == null) {
                throw new ValidationException("url is required and must be a valid URL");
            }
        } catch (URISyntaxException e) {
            throw new ValidationException("url is required and must be a valid URL");
        }
        if (request.createTime() == null) {
            throw new ValidationException("create_time is required");
        }
        if (request.createTime() < 0) {
            throw new ValidationException("create_time must be a non-negative epoch seconds integer");
        }
    }
}
```

**`IllegalStateException` on empty fetch after conflict:** If `findByUrl` returns empty after a `DataIntegrityViolationException`, something unusual occurred (row deleted between the insert attempt and the fetch). This is treated as a programming error, not a user error.

### 7. Tests — Testcontainers (unchanged pattern)

`@Testcontainers`, `@Container MySQLContainer<?>`, and `@DynamicPropertySource` added to `ItemRepositoryTest` and `ItemApiIntegrationTest` as specified in the original MySQL migration design. No additional test changes are required for the Hibernate switch — the integration tests exercise the full stack and remain dialect-agnostic.

## Critical Path Walkthrough

### Path 1: Store a new item (happy path)

1. Client POSTs `{"url": "https://example.com", "create_time": 1744367400}` to `POST /items`.
2. Jersey routes to `ItemController.create(ItemRequest)`.
3. `ItemService.create` validates url and create_time.
4. `ItemService` calls `repository.save(new Item(url, createTime))`.
5. `CrudRepository.save()` opens its own transaction; Hibernate issues `INSERT INTO items (url, create_time) VALUES (?, ?)` via HikariCP.
6. MySQL assigns `id`; Hibernate reads it back via `LAST_INSERT_ID()` (implicit with `IDENTITY` strategy). Transaction commits.
7. `save` returns the managed `Item` with its generated `id`.
8. `ItemService` returns `InsertResult(saved, true)`.
9. `ItemController` returns `201 Created` with the JSON-serialized `ItemResponse`.

### Path 2: Duplicate URL (idempotency)

1–4. Same as above.
5. `CrudRepository.save()` opens its own transaction; Hibernate issues the `INSERT`; MySQL rejects it with a unique-key violation. Transaction rolls back.
6. Spring wraps the JDBC `SQLException` in a `DataIntegrityViolationException`, thrown from `save()`.
7. `ItemService` catches it and calls `repository.findByUrl(url)`.
8. Hibernate issues `SELECT … WHERE url = ?`; returns the existing `Item`.
9. `ItemService` returns `InsertResult(existing, false)`.
10. `ItemController` returns `200 OK` with the original item's data.

### Path 3: MySQL unavailable during a write

1. Client POSTs to `POST /items`.
2. `ItemService.create` calls `repository.save(new Item(...))`.
3. HikariCP fails to acquire a connection within `connection-timeout` (3 seconds).
4. `SQLTransientConnectionException` propagates; no JAX-RS `ExceptionMapper` handles it.
5. Jersey returns `500 Internal Server Error`.
6. **No data loss** — the write never started.

## Tradeoff Log

| Decision | Options Considered | Choice | Rationale | Risks Accepted |
|---|---|---|---|---|
| **ORM layer** | JdbcTemplate, Spring Data JPA + Hibernate, jOOQ | Spring Data JPA + Hibernate | Removes manual SQL and `RowMapper` boilerplate; idiomatic for Spring Boot apps; `findByUrl` and `findAll` derived queries eliminate hand-written SQL entirely | Hibernate adds startup overhead and a learning curve for Hibernate-specific behavior (e.g. flush modes, N+1) |
| **Insert-or-fetch with JPA** | `INSERT IGNORE` via native query, try-`saveAndFlush`-catch in repository, try-`save`-catch in service | try-`save`-catch-`DataIntegrityViolationException` in `ItemService` | `CrudRepository.save()` is itself `@Transactional`, so it commits immediately when called without an outer transaction. Moving the try-catch to `ItemService` (which is not `@Transactional`) leverages this without requiring `JpaRepository.saveAndFlush` or an explicit `flush()` call | `ItemService.create` must never be annotated `@Transactional`; doing so would cause `save()` to join the outer transaction and defer the flush, breaking the catch. This is a non-obvious constraint. |
| **Repository interface** | `ItemJpaRepository` (package-private, injected into `ItemRepository` class), `ItemRepository extends CrudRepository` directly | `ItemRepository extends CrudRepository<Item, Long>` | Removes the two-interface indirection; `ItemRepository` is the single public contract and Spring Data implementation point | Try-catch logic cannot live in the interface, so it moves to `ItemService`, which now has a persistence concern (knowing to catch `DataIntegrityViolationException`) |
| **Entity / domain class** | Separate `ItemEntity` + `Item` record, single `Item` class with JPA annotations | Convert `Item` record to a JPA entity class | Eliminates the `ItemEntity` / `toDomain()` indirection; one class to maintain. Accessor names (`id()`, `url()`, `createTime()`) are preserved so no call sites change | `Item` is no longer immutable; callers must not mutate it after fetch. The `domain` package now has a JPA dependency (`jakarta.persistence`) |
| **`ddl-auto` setting** | `create`, `update`, `validate`, `none` | `validate` | Flyway is the schema authority; Hibernate should only confirm the mapping is consistent with the DB, not modify it | Hibernate validation runs on every startup — a schema/entity mismatch causes a startup failure. This is intentional. |
| **`open-in-view`** | Default (`true`), `false` | `false` | Holding a DB connection open for the full HTTP request lifecycle wastes pool connections unnecessarily on a single-user service | None for this workload; notable if lazy-loaded associations were added later |
| **MySQL version** | MySQL 8.0, MySQL 8.4 LTS, MySQL 9.x Innovation | MySQL 9.2 | Latest release; user explicitly requested 9.x | Innovation releases move faster than LTS; watch for deprecations between minor versions |
| **URL uniqueness** | Full-column unique, prefix index, SHA-256 hash column | Prefix index `url(255)` | Simplest approach; edge case (two URLs identical in first 255 chars) is not realistic for a reading list | Theoretical collision on very long, similar URLs |
| **Schema migrations** | `schema.sql` + Spring init, Flyway, Liquibase | Flyway | Versioned migrations are required as soon as the schema evolves; Spring Boot auto-configures it with zero boilerplate | Flyway history table must be bootstrapped on a fresh DB; the `listerizer` user needs `CREATE` privilege |
| **Test strategy** | H2 (MySQL-compat mode), Testcontainers, dedicated test MySQL | Testcontainers | Runs real MySQL 9.2; eliminates dialect divergence; Flyway runs in test context automatically | First test run downloads the Docker image; CI must have Docker available |
| **Connection pool size** | 1 (SQLite legacy), 5, 10 (HikariCP default) | 5 | Single-user load; 5 handles burst without over-provisioning MySQL connections | Under unexpected concurrent load, pool could exhaust; increase if needed |

## Operational Concerns

### Provisioning MySQL

MySQL must be running and the `listerizer` database + user must exist before the application starts. Recommended setup:

```sql
CREATE DATABASE listerizer CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
CREATE USER 'listerizer'@'%' IDENTIFIED BY '<password>';
GRANT SELECT, INSERT, UPDATE, DELETE, CREATE ON listerizer.* TO 'listerizer'@'%';
```

`CREATE` is required for Flyway to create its `flyway_schema_history` table on first startup. It can be revoked afterwards if a tighter security posture is desired, but Flyway will need it again whenever it runs against a fresh database.

### Environment Variables

| Variable | Default | Required |
|---|---|---|
| `MYSQL_HOST` | `localhost` | No |
| `MYSQL_PORT` | `3306` | No |
| `MYSQL_DB` | `listerizer` | No |
| `MYSQL_USER` | `listerizer` | No |
| `MYSQL_PASSWORD` | _(none)_ | **Yes** |

`MYSQL_PASSWORD` has no default and must be supplied. Spring Boot will fail to start if it is absent.

### Docker (local dev)

Add a `docker-compose.yml` for running MySQL alongside the app locally:

```yaml
services:
  mysql:
    image: mysql:9.2
    environment:
      MYSQL_ROOT_PASSWORD: root
      MYSQL_DATABASE: listerizer
      MYSQL_USER: listerizer
      MYSQL_PASSWORD: listerizer
    ports:
      - "3306:3306"
    volumes:
      - listerizer-mysql:/var/lib/mysql

  app:
    build: .
    ports:
      - "5080:5080"
    environment:
      MYSQL_HOST: mysql
      MYSQL_PASSWORD: listerizer
    depends_on:
      - mysql

volumes:
  listerizer-mysql:
```

### Kubernetes

Replace the SQLite `PersistentVolumeClaim` in the existing manifest with a MySQL `Deployment` + `Service`, or point `MYSQL_HOST` at an external managed MySQL instance (preferred for production). The app `Deployment` spec removes the `volumeMount` and `volume` blocks and adds the MySQL env vars.

### Backup

Use `mysqldump`:

```bash
mysqldump -h $MYSQL_HOST -u listerizer -p listerizer items > listerizer-$(date +%Y%m%d).sql
```

### Monitoring

No new monitoring required beyond what is already planned. The existing `500` on DB unavailability is observable in systemd/Kubernetes logs. A `GET /health` endpoint returning MySQL connectivity status is a natural follow-up.

## Out of Scope / Future Work

- **Data migration**: Moving existing SQLite rows to MySQL is a separate one-time task.
- **Read replicas**: Single-user load does not justify a replica. Add when read latency becomes a concern.
- **Additional Flyway migrations**: The initial `V1` migration is in scope. Future migrations (e.g. `V2__add_read_at.sql`) are authored as the schema evolves.
- **Connection retry / circuit breaker**: A `503` response on MySQL unavailability is better than a raw `500`. Defer until the service leaves localhost.
- **SSL/TLS for MySQL connection**: Required if MySQL is not on the same host or private network. Add `sslMode=REQUIRED` to the JDBC URL at that point.
- **Lazy-loaded associations**: Not applicable for the current single-entity model; if added later, `open-in-view=false` means associations must be fetched within the transaction boundary.
